case study · enterprise-network-platform

A private motorway network for 5,000 people — proofread before a single road is built.

Large companies run their own private road network for data — offices, a data center, and cloud regions that all need to reach each other reliably. This project builds one of those networks in miniature, and proves its routing logic is sound before a single switch is powered on.

# proofread first Every change to this network's configuration is checked by an automated inspector before it's ever deployed — for routing loops, broken references, mismatched sessions — on paper, in minutes. Open the proofreader in the diagram below for how that works.

the idea, without the jargon

$ how to think about this

Think of this as the road network for a company of 5,000 people, spread across a headquarters, a branch office, and two AWS regions. There are motorways between the major sites, junctions where local traffic joins the highway, and a border checkpoint where traffic leaves the country entirely for the cloud.

Every junction has to agree on the rules — which way is fastest, what happens if a road closes, how a car from the branch office finds its way to a server sitting in AWS. That agreement is negotiated automatically, the moment two routers connect to each other.

And before a single road is actually built, an inspector proofreads every road sign on paper — checking for signs that contradict each other, or point nowhere — so mistakes are caught in the blueprint, not on opening day.

what could go wrong

$ failure modes, handled on purpose

If this happensWhat the system does
One of the two spine switches goes down Bandwidth halves as one path disappears, but the fabric stays reachable — a health check notices in under a second.
A single leaf switch goes down Only the hosts on that one leaf lose connectivity; the fix is restarting it and reprovisioning its config.
The WAN link to headquarters goes down The branch office and the data center become isolated from each other — no second path exists yet in this model.
The cloud-edge router goes down The hybrid tunnel to AWS drops, but the data center itself keeps running unaffected.
An entire AWS availability zone fails Cloud resources fail over automatically to another zone; the Transit Gateway itself is resilient across zones.
# what this project is not: a network carrying real traffic. All 5,000 users are simulated, the AWS side of Terraform is manual-dispatch only with no live account assumed, the proofreader checks routing logic only — not the actual forwarding hardware — and there's no second WAN path yet if the link to headquarters fails.
# glossary — 10 terms, click to expand
EVPN-VXLAN
A way to make servers on different physical switches act like they're on one local network, by wrapping their traffic in an extra header.
eBGP
A routing protocol used between the data-center switches here, chosen because it scales cleanly to hundreds of switches without loops.
OSPF
A routing protocol used across the office and branch network, so every router learns the fastest path to everywhere else.
IKEv2
The protocol that negotiates and secures the encrypted tunnel connecting the office network to AWS.
Transit Gateway
A central AWS router that every cloud network connects to, instead of connecting to each other directly.
Batfish
A tool that reads network configuration files and calculates what the network would actually do — without needing it to be running.
pyATS
A testing framework that logs into real, running network devices and checks they behave the way they were designed to.
Route reflector
A device that shares routing information with many others, so they don't all need to connect to each other directly.
ASN
An ID number for a chunk of network under one set of routing rules — every major site here has its own.
ECMP
Splitting traffic across multiple equal paths at once, so losing one path only reduces capacity instead of cutting it off.
