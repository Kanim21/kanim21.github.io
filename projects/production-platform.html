case study · production-platform

Infrastructure built for Black Friday traffic, not an average Tuesday.

An online store's traffic can jump ten-fold in an hour when a sale goes live, and a slow checkout page directly costs revenue. This project designs the servers, network, and database underneath a store to survive that spike, fail predictably when something breaks, and rebuild itself from version-controlled blueprints rather than manual clicks.

# reference architecture This is a portfolio piece, not a live account — terraform apply is wired but gated behind a manual trigger, because running the real thing 24/7 costs roughly $700–900/month for a project with no live customers. Everything else — the lint, the security scan, the design — is real.

the idea, without the jargon

$ how to think about this

Imagine designing a physical shop knowing that one day a year, ten times the usual number of customers will show up at once. You wouldn't build for a normal Tuesday and hope — you'd plan wider doors, more till staff on call, and a stockroom that can restock itself.

This project is that shop, built in code. A doorman sends each customer to a free till. The till staff are backed by a stockroom that keeps a single, trustworthy ledger of every order. When the crowd doubles, more till staff clock in automatically, no manager required.

And the whole shop — doors, tills, stockroom layout — exists first as a blueprint, not bricks. That blueprint lives in version control, gets checked by an inspector before anything is built from it, and can be rebuilt identically if it ever needs to be.

what could go wrong

$ failure modes, and the recovery time

If this happensWhat the system does
An EKS node stops responding Cluster Autoscaler replaces it and pods reschedule automatically — typically back to normal within 5 minutes.
An entire availability zone goes down The load balancer drains it and Kubernetes reschedules everything to the remaining zones — under 10 minutes.
The database's primary instance fails Aurora auto-promotes a standby and DNS flips to it — about 30 seconds for the database, under 5 minutes end-to-end.
A bad deploy ships A rollout-stalled alert fires; an engineer rolls it back with one command — typically resolved within 5 minutes.
Secrets are suspected leaked GuardDuty flags it; secrets rotate in Secrets Manager and pods pick up new credentials on restart — under 30 minutes.
# what this project is not: a live, deployed system — there's no AWS account behind it, since running it 24/7 costs roughly $700–900/month for a project with no real customers. Redis caching is designed but not yet wired up, and database secret rotation is a manual runbook, not yet automatic.
# glossary — 10 terms, click to expand
CDN
A network of servers spread around the world that cache and serve content close to the visitor, instead of one distant server handling everyone.
VPC
A private, isolated network inside a cloud provider — like renting your own building instead of sharing office space.
Kubernetes / EKS
A system that runs an application as many small containers across a fleet of servers, restarting or replacing them automatically.
Multi-AZ
Running duplicate copies of something in physically separate data centres, so one location failing doesn't take the whole system down.
OIDC
A way for one system (like a CI pipeline) to prove its identity and get a short-lived credential, without a stored password.
IRSA
The same idea as OIDC, but for an individual running pod requesting AWS permissions instead of a whole CI job.
IaC (Infrastructure as Code)
Describing servers and networks in text files instead of clicking through a console, so the setup can be reviewed and rebuilt exactly.
RTO
Recovery Time Objective — the target maximum time a system is allowed to be broken before it's considered recovered.
Autoscaling
Automatically adding or removing running copies of a service based on how much traffic or load it's currently under.
Static analysis
Checking code or configuration for known mistakes by reading it, without actually running it.
