case study · omni-obs-platform

Three simulated regions, one shared view — and an alarm that only rings when it should.

Every region of a company's infrastructure produces its own flood of numbers, and normally nobody can see all of it in one place at once. This project builds that single place, and teaches it to stay quiet until a trend actually means trouble.

# the core idea This system doesn't promise zero failure — nobody can. It promises a budget for how much failure is acceptable, and watches how fast that budget is being spent. Open the early-warning system in the diagram below for how that actually works.

the idea, without the jargon

$ how to think about this

Picture a hospital's vital-signs monitors — one at every bed, in every ward, quietly measuring heart rate and blood pressure all day. Nobody expects a nurse to walk to every single bed to check on every patient every minute.

Instead, everything reports back to one central nurses' station. A screen there shows the whole ward at a glance, and — crucially — it doesn't beep for every tiny fluctuation. Some variation is normal; the station only raises an alarm when a trend actually looks dangerous.

This project builds that pattern for computer systems spread across three regions instead of three wards. Each region keeps working even if its line to the station drops, and the station combines all three into one screen, with alarms tuned to only interrupt someone for a trend that's actually burning through the failure the team agreed was acceptable.

what could go wrong

$ failure modes, handled on purpose

If this happensWhat the system does
One region's connection to the central station goes down That region keeps collecting and storing locally on its own — nothing is lost, only the merged view is briefly missing that region.
Errors climb fast, burning the failure budget at 14.4× the normal rate A page fires within minutes, pointing straight at a written runbook with the likely cause and severity.
Errors climb slowly over hours instead of spiking A lower-urgency ticket fires instead of a page — same problem, not urgent enough to wake anyone at 3am.
The central query service gets overwhelmed by a very large question It can run out of memory and restart — the documented fix is raising its memory limit, not silently failing forever.
Someone edits the live cluster by hand instead of the git repo The GitOps controller notices the drift and quietly reverts it back to whatever the repository says should be running.
# what this project is not: a production observability platform. All three regions run on one laptop's Docker engine, the traffic is synthetic, only two hours of history is kept without extra storage wired in, and Grafana's login is a shared demo password, not a real identity system.
# glossary — 10 terms, click to expand
OTLP
The common data format most modern monitoring tools use to send metrics, so a collector doesn't need custom code per app.
SLO
A Service Level Objective — a target for how reliable something should be, agreed on in advance, not decided after an incident.
Error budget
The small amount of allowed failure inside an SLO — 99.9% reliable leaves a 0.1% budget to spend before anyone panics.
Burn rate
How fast that failure budget is being spent right now, compared to a steady, sustainable pace.
GitOps
A way of running infrastructure where the git repository is the plan, and a robot continuously makes the real system match it.
Federation
Combining data from several independent systems into one unified view, without merging the systems themselves.
Kustomize
A tool for describing infrastructure once, then layering small, explicit differences on top for each place it runs.
Sidecar
A helper process that runs alongside a main one, handling a side task — here, exposing data to the outside world.
Object storage
Cheap, durable storage for large files or data blocks — used for long-term history instead of a live database.
Reconciliation
Comparing the real, running state of a system against the intended state, and fixing any difference found.
