case study · pipeline-surgeon-labs

A mechanic who diagnoses a broken build and proposes a fix — without touching the car.

When a software build breaks, someone has to read pages of error logs to figure out why — usually at the worst possible time. This project reads the log itself, diagnoses the failure, and leaves a written repair proposal for a human to approve, never touching the code directly.

# by the numbers Across all six known test failures, this agent classified every one correctly — 100% accuracy — at about $0.05 and 21 seconds per run. It never merges anything itself; every proposed fix waits in a pull request for a human.

the idea, without the jargon

$ how to think about this

Picture a mechanic who reads your car's fault codes without popping the hood. They look at what the engine reported, work out what's actually wrong, and write it up on a clipboard — they don't start pulling parts.

Before writing anything down, they double-check the real part number instead of guessing from memory. A diagnosis built on a wrong part is worse than no diagnosis at all.

Then they leave the clipboard on the counter and walk away. They never touch the car. A person reads the note, decides whether the diagnosis makes sense, and only then picks up a wrench.

what could go wrong

$ failure modes, handled on purpose

If this happensWhat the system does
The build log contains a real AWS key or GitHub token The redactor strips it via pattern-matching before the log ever leaves the runner or reaches Claude.
Claude isn't confident in its diagnosis (below 0.6) The agent stops and opens a diagnostic pull request explaining why, instead of proposing a fix it doesn't trust.
The proposed diff doesn't apply cleanly to the code The patch step fails, the agent exits with an error, and no pull request claiming a fix ever opens.
The agent hits its 5-round tool-use budget with no diagnosis It returns a hard-coded "unknown" result at 0% confidence rather than looping forever or guessing.
The classification or the fix is simply wrong A human reviewing the pull request catches it — nothing merges to `main` without that review.
# what this project is not: an autonomous CI fixer. It never commits to `main`, never retries a failed patch, and only recognises three Maven failure types — anything else comes back "unknown." Proven against six captured fixtures, not a large real-world sample.
# glossary — 10 terms, click to expand
Maven
A build tool for Java projects. When it fails, this whole system kicks in.
Unified diff
A standard text format showing exactly which lines changed, so a human can review a fix at a glance.
Tool use (LLM)
Letting an AI call a predefined function to get real data, instead of answering from memory alone.
Confidence score
A 0–1 number the model attaches to its own answer, used here to decide whether to trust it.
Redaction
Automatically blacking out things like passwords or API keys from text before it's sent anywhere.
Pull request
A proposed code change, visible and reviewable, that a human must approve before it becomes real.
CI
Continuous Integration — automatically building and testing code every time someone pushes a change.
Fixture
A saved, real example — here, a captured failing build log — used to test that software still behaves correctly.
Token budget
A hard cap on how much text an AI model can read or write in one run, used to control cost.
Classification
Sorting something into one of a fixed set of known categories — here, three failure types, or "unknown."
